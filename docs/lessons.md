# Lessons

## 2026-03-06
- Runtime/protocol work in `agtmux-term` must be closed on fresh post-fix verification only. Focused tests are not enough if a later fix lands; rerun the relevant build/tests after the final patch and record only that fresh result in `docs/70_progress.md`.
- `SACSetScreenSaverCanRun returned 22` can appear during macOS targeted UI test runs even when the test passes. Treat it as a non-fatal environment warning unless accompanied by an actual test failure.
- Review routing policy can be asymmetric by tool: Claude review may require the real Claude Code CLI, while Codex review can be satisfied by Codex subagents if that is the explicit repository/user policy. When Claude does not return a usable verdict, increase Codex review coverage instead of fabricating a Claude result.
- For exact-target Workbench `Rebind`, preserve the existing tile identity so focus/layout anchors remain stable. Only clear hint-only terminal `SessionRef` fields when the exact target/session changes, and keep document pinning unchanged across rebind.
- When delegated implementation tooling is unstable, use an explicit escalation ladder instead of stalling: real agent CLI first, then Codex subagent, and finally orchestrator direct execution only after both delegated paths fail to produce a usable response.
- When a user corrects an environment diagnosis, replace the stale blocker in tracking/docs immediately and re-check current evidence before repeating the old explanation. In this repo, `screenLocked=1` from an earlier XCUITest rerun was not sufficient evidence for a later `testPaneSelectionWithMockDaemonAndRealTmux` skip.
- For tmux-driven UI harnesses, do not assume `tmux -F` interprets escape sequences like `\t`; prefer a literal delimiter such as `|` and parse exactly what tmux emits. For dynamic terminal retarget state, prefer an app-state snapshot oracle over AX `value` when XCTest does not expose that field reliably.
- In pane-selection UI proofs, keep accessibility identity and canonical model identity separate. `AccessibilityID.paneKey(...)` is for UI lookup, while the canonical active selection contract is `AgtmuxPane.id` (`source:session:pane`). Mixing them creates false-red E2E failures even when live tmux targeting is correct.
- For main-panel pane-retarget regressions, a UI proof that replaces Ghostty with a UITest placeholder is structurally insufficient. The trusted E2E contract must exercise a real terminal surface and assert rendered attach state, not just sidebar/store/tmux intent.
- When Xcode-targeted proof matters in this repo, regenerate the project before trusting `xcodebuild` results after adding or renaming source files. `swift build` can pass while the checked-in `AgtmuxTerm.xcodeproj` is still missing a newly added file such as `GhosttySurfaceTelemetryBridge.swift`.
- When the user explicitly switches a bugfix slice to orchestrator-owned implementation, stop spending turns on real-agent implementation delegation. Update `AGENTS.md` and tracking docs first, then move straight into TDD and verification.
- Same-session pane retarget and initial attach should not share one global desired-convergence threshold. In this repo, initial attach still needs a stricter confirmation window, but retargeting an already observed rendered client must release stale desired state after the first matching observation so terminal-originated reverse sync is not overwritten.
- When a task reopens and recloses multiple times in one day, reconcile `docs/65_current.md` and the `docs/70_progress.md` summary immediately after the final green rerun. Leaving stale blocker text in current-state docs causes the next debugging pass to start from an already-invalid assumption.
- For `NSTextInputClient` on macOS, do not send raw terminal key events before `interpretKeyEvents(...)`. If Enter/Return reaches the terminal path first, Japanese IME commit can be eaten as a raw newline and never reach the shell as confirmed text.
- For cross-repo live E2E, do not duplicate the same ownership in both repos. In this project pair, `agtmux` owns producer-side semantic truth for real CLI scenarios, while `agtmux-term` owns thin daemon-to-sidebar canaries that use exact daemon payload truth as the primary oracle.
- For macOS titlebar icon controls, XCUITest does not reliably expose the parent button's AX `value`. If a badge count matters, expose a dedicated badge child with a stable identifier and keep aggregate count truth in integration tests rather than forcing the UI proof to rely on the button value.
- Fresh-socket online E2E is not enough for daemon-to-app compatibility when the shipped app uses a persistent daemon state DB. If producer truth can carry orphan managed rows from old sessions, add a dirty-state regression or explicit live smoke against the persistent app-managed socket; otherwise strict-consumer breakage can slip past all clean-socket online tests.
- When validating a daemon-side protocol fix against the shipped app path, restart the persistent app-managed daemon before judging the result. A clean rebuilt binary is not enough if the long-lived process serving `/Users/virtualmachine/Library/Application Support/AGTMUXDesktop/agtmuxd.sock` has not been restarted onto the fixed bootstrap logic yet.
- If macOS blocks XCUITest before the test body with `Timed out while enabling automation mode.`, do not treat that as product evidence and do not leave recurrence protection dependent on that runner. Add or strengthen a lower-layer live canary at the app-consumer boundary for the same symptom, then keep the XCUITest as an optional visibility proof.
- Sandbox-isolated `xctrunner` processes in this repo do not inherit a reliable shell environment. Normalize runner PATH explicitly in UITest shell helpers, and treat `codex login status: Not logged in` from the runner as a separate auth/environment issue from automation-mode permission.
- In this repo, sandboxed `xctrunner` cannot be trusted as the oracle for real CLI auth availability. If the actual app path runs unsandboxed and lower-layer live canaries already prove real Codex/Claude auth, do not block metadata-enabled UI proofs on runner-side `codex login status`; let the UI test exercise the app path and treat later activation/background failures as a separate harness issue.
- When an upstream daemon semantic fix lands and lower-layer live canaries go green, close the product task immediately and split any remaining XCUITest failure into its own harness task. Do not keep describing a `Running Background` foreground-activation failure as if it were still part of daemon truth or runner auth.
- When local tmux inventory already exists, `ui.bootstrap.v2 panes=[]` is not a ready sync-v2 epoch. If the consumer primes exact-row ownership on that empty bootstrap, later change replay has no visible `session_name` / `window_id` mapping and cannot recover managed overlay. In this repo, treat that state as startup-not-ready, reset sync ownership, and retry.
- Do not make live UI proofs depend on hidden accessibility descendants underneath a `.combine` row wrapper. In this repo, managed pane provider/activity/freshness must be exposed on the visible pane row itself via a stable row-level AX contract; otherwise AppViewModel/sidebar truth can be green while XCUITest still cannot discover the metadata.
- For real-Codex live canaries, do not assume plain completion implies product-facing `waiting_input`. Sync-v2/list/json may still say `waiting_input`, but sync-v3 product truth intentionally emits `completed_idle` unless there is an explicit pending user-input request. Keep `waiting_input` / attention coverage deterministic unless the live lane proves real `pending_requests`.
